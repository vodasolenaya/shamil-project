# ТЗ: Telegram-бот + 7-дневная воронка утепления после диагностики

> Задача для Claude Code. Контекст проекта — в `PROJECT_CONTEXT.md` в корне проекта. Там же — кейсы учеников, голос бренда, ЦА.

---

## 1. Зачем это вообще делается

Сейчас воронка artofsales.art:

```
Трафик → / или /diagnostic → форма → Шамиль пишет в ТГ руками → созвон → продажа
```

Узкое место: человек прошёл диагностику, но **не оставил телефон / не записался на созвон / не заплатил 990₽**. Он просто ушёл. В PROJECT_CONTEXT указано: «80+ фрилансеров прошли диагностику» — при 47 купивших это значит ~40% конверсия в программу от тех, кто дошёл до диагностики. Остальные 60% сейчас ничем не греются.

Воронка утепления в боте превращает «ушедших» в «вернувшихся». У топовых инфобизнесменов (Пырикков) такая бот-воронка делает 20-40% доп. продаж от холодной базы.

**Цель:** автоматическая 7-дневная цепочка сообщений в `@artofsales_shamil_bot`, которая забирает человека после диагностики, греет его кейсами/смыслами, закрывает возражения и возвращает к Шамилю на созвон.

---

## 2. Что уже есть

- `diagnostic.html` — 8 открытых вопросов. После отправки формы на `/thanks.html` предлагается подписаться на бота: `@artofsales_shamil_bot?start=quiz`
- Бот `@artofsales_shamil_bot` создан, но backend пустой. Токен есть у Шамиля.
- `api/` директория в проекте (похоже, куда деплоилась форма)
- Деплой: Vercel

## 3. Чего нет, что нужно сделать

1. **Backend для бота** — принимать webhook от Telegram, понимать `/start quiz`, записывать лида с привязкой к его ответам на диагностике
2. **БД** для лидов + их ответов + статуса воронки
3. **Связка diagnostic.html ↔ бот** — чтобы бот знал, кто пришёл, и имел его ответы на 8 вопросов
4. **Scheduler** — отсылает сообщения N-го дня в назначенное время
5. **Логика остановки** — если человек ответил «интересно/запишусь» / уже купил / попросил «не пиши больше»
6. **Notification Шамилю** — алерт, когда человек отвечает на сообщение бота, чтобы он мог вклиниться лично

---

## 4. Архитектура (рекомендация, можно менять)

**Стек:** Node.js + Telegraf (или Python + aiogram — выбери что удобнее), Postgres (через Neon / Supabase / Vercel Postgres), cron job на Vercel Cron.

**Почему не serverless-only:** Telegram-боту нужен webhook-endpoint, который стабилен. Vercel Functions ок, если хранить очередь в БД и запускать scheduler по cron.

**Минимальный путь:**

```
1. diagnostic.html отправляет ответы → POST /api/submit-diagnostic
2. /api/submit-diagnostic:
   - пишет в БД таблицу `leads` (id, tg_handle, quiz_answers_json, created_at)
   - возвращает deep-link: https://t.me/artofsales_shamil_bot?start=<lead_id>
3. Пользователь жмёт на кнопку «Подписаться на бота» → бот получает /start <lead_id>
4. Бот:
   - связывает telegram_user_id с lead_id в БД
   - создаёт в таблице `drip_schedule` 7 записей (send_at, step, message_key)
   - отправляет Сообщение #0 (приветствие)
5. Cron каждый час проверяет `drip_schedule` где send_at <= NOW() AND sent_at IS NULL
   → отправляет сообщение, помечает sent_at
6. Любое сообщение от пользователя в бот → пересылается Шамилю в ЛС через бот
   + в базе ставит drip_schedule.paused = true (чтобы не заваливать)
7. Шамиль из ЛС может через команду /resume <user_id> вернуть в воронку если нужно
```

**Таблицы:**

```sql
leads (
  id uuid pk,
  tg_user_id bigint nullable,   -- заполняется при /start
  tg_handle text nullable,
  quiz_answers jsonb,            -- {q1: "80к, хочу 200к", q2: "...", ...}
  source text,                   -- "diagnostic" | "paid-990" | "manual"
  status text,                   -- "new" | "active" | "engaged" | "converted" | "unsubscribed"
  created_at timestamptz,
  converted_at timestamptz nullable
)

drip_schedule (
  id uuid pk,
  lead_id uuid fk,
  step int,                      -- 1..7
  send_at timestamptz,
  sent_at timestamptz nullable,
  paused boolean default false,
  message_key text               -- ключ в messages.json
)

events (
  id uuid pk,
  lead_id uuid fk,
  type text,                     -- "message_sent" | "user_replied" | "paused" | "unsubscribed"
  payload jsonb,
  created_at timestamptz
)
```

**Расписание по умолчанию (МСК):**

| Шаг | Когда | Время |
|---|---|---|
| 0 | сразу после /start | моментально |
| 1 | +1 день | 11:00 |
| 2 | +2 дня | 11:00 |
| 3 | +4 дня | 18:00 |
| 4 | +6 дней | 11:00 |
| 5 | +8 дней | 18:00 |
| 6 | +10 дней | 11:00 |
| 7 | +12 дней | 18:00 |

(не «7 дней подряд» — это слишком навязчиво; разрыв между касаниями растёт)

## 6. Персонализация и логика

**{{name}}** — имя из формы диагностики. Если нет — `"привет"`.

**{{quiz_q5_short}}** — первые 120 символов ответа на Q5 диагностики («Что тебя останавливает поднять ценник прямо сейчас?»). Добавить в начало "> " для цитаты.

**Маршрут ответа пользователя в боте:**

```
если ответ содержит ["да", "интересно", "хочу", "запиши", "разбор", "созвон", "когда", "сколько", "цена"]:
  → pause drip
  → notify Шамиля с текстом и ссылкой на lead в админке
  → Шамиль отвечает лично из своего аккаунта (бот форвардит)

если ответ содержит ["стоп", "не пиши", "отпишись", "хватит", "unsubscribe"]:
  → status = "unsubscribed"
  → drip cancelled
  → Шамилю ничего

иначе:
  → pause drip на 24ч
  → notify Шамиля
  → через 24ч если ответа нет — resume drip
```

## 12. Критерии готовности

1. Человек проходит `/diagnostic` → заполняет форму → видит кнопку «Открыть бота» → жмёт → бот приветствует лично по имени → через 24 часа приходит сообщение #1.
2. Я отвечаю в боте «да» → Шамиль получает уведомление с моим именем и текстом ответа.
3. Я пишу «стоп» → сообщения больше не приходят.
4. В админке видны все мои шаги, можно вручную поставить пауза/resume.
5. Сообщения отправляются в указанное время МСК, не в 3 часа ночи.

---

**Приоритет:** HIGH. Это единственное узкое место воронки, которое недобирает при уже работающем трафике. Остальные задачи (новый ЛМ, серия постов по болям, закрытая презентация) — после того, как этот контур закрыт.
